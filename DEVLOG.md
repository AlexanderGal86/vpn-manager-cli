# DEVLOG — Журнал разработки VPN / Proxy Manager

Хронологическая история решений, проблем и архитектурных выборов.

---

## 2025-03-18 · Сессия 1: Планирование и архитектура

### Исходные требования
Пользователь запросил CLI-инструмент из трёх независимых модулей:
1. Сборщик IP из открытых источников (HTML, файлы, API) с TCP-пингом
2. Тестировщик скорости (батчи по 10, параллельно по 2, порог YouTube)
3. Коннектор Windows с системным прокси и CLI-меню

**Ключевые решения архитектуры:**

- Язык: Python 3.10+ — кроссплатформенный, хорошая стандартная библиотека,
  `winreg` встроен для Windows
- Формат обмена между модулями: JSON (структурированный, читаемый человеком,
  лёгкий в отладке). CSV и TXT отклонены — теряют метаданные (тип, время проверки)
- Три папки модулей + `main.py` для pipeline — модули независимы,
  но связаны через файлы в `output/`
- ANSI-цвета через escape-коды напрямую (не colorama) — меньше зависимостей,
  Windows 10+ поддерживает через `SetConsoleMode`

### Структура выходных данных
```
proxy_list.json: [{ip, port, type, ping_ms, status, checked_at}]
viable_proxies.json: [{...+ speed_mbps, latency_ms, test_status, tested_at, fail_count}]
```

## 2025-03-18 · Сессия 2: Реализация Module 1

### Проблема: разнообразие структур HTML-таблиц

Разные сайты используют разные CSS-классы и порядок колонок.
**Решение**: конфигурационные словари `HTML_SOURCES` с полями `ip_col`, `port_col`,
`type_col`, `https_values`. Парсер generic, конфигурация декларативна.

### Проблема: ICMP ping требует root на Linux

`ping3` библиотека требует `sudo` для raw sockets.
**Решение**: TCP-пинг через `socket.connect()` — не требует прав,
работает везде, измеряет реальное время установки соединения.

### Архитектурный выбор: ThreadPoolExecutor vs asyncio

Asyncio потребовал бы `aiohttp` и `asyncio`-aware код во всех функциях.
ThreadPoolExecutor из стандартной библиотеки, проще, достаточно для 40 воркеров.
Для I/O-bound задач (сетевые запросы) GIL не проблема.

### Выбор источников

- `free-proxy-list.net` и `sslproxies.org` — стабильные, простая структура таблиц
- `ProxyScrape API` — надёжный, возвращает большие объёмы, три протокола
- `TheSpeedX/PROXY-List` на GitHub — обновляется часто, простой формат
- `GeoNode API` — структурированный JSON, есть пагинация

## 2025-03-18 · Сессия 3: Реализация Module 2

### Проблема: как показать прогресс 2 параллельных тестов

Нужно обновлять одну строку CLI для каждого из двух параллельных прокси.
**Решение**: `threading.Lock` + общий словарь `progress = {ip: status_string}`,
обновляется в коллбэке каждого future, выводится через `\r` overwrite.

### Выбор тестового URL для скорости

Требования: открытый, без авторизации, стабильный, файл ≥ 2МБ.
- `speedtest.tele2.net` — шведский CDN, хорошее глобальное покрытие
- `ipv4.download.thinkbroadband.com` — UK, альтернатива
- `speedtest.ftp.otenet.gr` — греческий, третий fallback

**Решение fallback**: цикл `for url in SPEED_TEST_URLS: try: ... break`

### Порог YouTube

- SD 480p: ~1-2 Мбит/с
- HD 720p: ~2.5 Мбит/с
- HD 1080p: ~5 Мбит/с
- 4K: ~20 Мбит/с

Выбрано `MIN_SPEED_MBPS = 5.0` — достаточно для HD без буферизации.

## 2025-03-18 · Сессия 4: Реализация Module 3 (Windows Proxy)

### Почему три метода установки прокси?

**winreg** (основной): самый быстрый, прямой доступ к реестру без внешних процессов.
Работает на любой Windows с Python.

**PowerShell** (fallback): на некоторых корпоративных машинах с групповыми политиками
прямой `winreg` может быть ограничен. PowerShell часто имеет другой уровень доступа.

**netsh** (дополнение): WinINet (браузеры) и WinHTTP (системные службы) — два
разных стека. Реестр меняет только WinINet. `netsh winhttp import proxy source=ie`
синхронизирует WinHTTP с настройками IE/WinINet.

### SOCKS в Windows системном прокси

Формат `socks=ip:port` — недокументированное, но работающее расширение WinINet.
Chrome, Edge, Firefox читают именно это поле.
SOCKS5 и SOCKS4 через эту настройку неразличимы для Windows —
разница обрабатывается на уровне приложения.

### ProxyOverride

Без этого поля локальные адреса идут через прокси → ломаются локальные сервисы,
localhost веб-серверы, корпоративные ресурсы. Включаем стандартный набор:
`localhost;127.*;10.*;172.16.*;192.168.*;<local>`

## 2025-03-18 · Сессия 5: Дистрибуция

### setup.bat: почему не просто ZIP?

ZIP требует: распаковать → найти папку → открыть CMD → `python install.py`.
BAT: двойной клик → всё само.

**Проблема**: `cmd.exe` имеет лимит ~8191 символов на строку SET-переменной.
Base64 архива ~87K символов. Решение — PowerShell heredoc с `$b64 += "chunk"`.
PowerShell не имеет такого лимита.

**Проблема**: кодировка BAT-файла. Windows cmd по умолчанию CP866/CP1251.
Решение: писать файл в UTF-8 с `\r\n` (Windows line endings), избегать
кириллицы в BAT-командах (только в `echo`).

### setup.exe: PyInstaller

PyInstaller собирает всё в один EXE: Python runtime + stdlib + зависимости.
Размер ~7МБ — приемлемо для installer.

**Ограничение**: EXE собирается под текущую ОС. Linux → Linux ELF, Windows → Windows PE.
Решение: дать пользователю `build_exe_source.py` + `build_exe_on_windows.bat`
для сборки на его машине.

## 2025-03-18 · Сессия 6: Умный перезапуск

### Проблема: при повторном запуске теряется уже собранная база

Первый запуск собирает 500+ адресов, проверяет пинг 5 минут.
Повторный запуск через час — всё сначала. Нерационально.

**Решение — TTL-кэш в Module 1**:
- Адреса с `checked_at` < 6 часов → не перепинговать (они ещё свежие)
- Адреса старше → перепроверить (могли умереть)
- Новые адреса из источников → добавить и проверить

**Решение — перепроверка в Module 2**:
- Viable прокси могут деградировать со временем
- TTL 3 часа на результаты тестов скорости
- Счётчик неудач: прокси, который 3 раза подряд не прошёл тест → удалить

### Проблема: datetime без timezone (timezone-naive)

Старые записи в JSON не имели timezone в `checked_at`.
`datetime.fromisoformat("2024-01-15T10:30:00")` возвращает naive datetime.
Сравнение naive с aware (`datetime.now(timezone.utc)`) → `TypeError`.

Решение:
```python
if checked.tzinfo is None:
    checked = checked.replace(tzinfo=timezone.utc)
```

Попутно переход с `datetime.utcnow()` (deprecated в 3.12) на
`datetime.now(timezone.utc)`.

## Технический долг и известные ограничения

| Проблема | Статус | Приоритет |
|---|---|---|
| TCP-пинг ≠ HTTP-доступность прокси | Известно, Module 2 доделывает | Низкий |
| EXE только для Windows (PyInstaller) | Известно, задокументировано | Низкий |
| SOCKS5/4 неразличимы в WinINet | Известно, Windows ограничение | Низкий |
| Нет retry при временных сетевых ошибках в Module 1 | Открыто | Средний |
| Нет proxy rotation в Module 3 | В roadmap | Средний |
| HTML-источники могут сломаться при изменении верстки | Известно | Высокий |
